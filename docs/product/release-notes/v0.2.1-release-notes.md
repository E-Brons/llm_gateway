# Release Notes — v0.2.1

**Date:** 2026-04-11
**Status:** Release

---

## Overview

v0.2.1 is a bug-fix patch for the `ipadapter_faceid` diffusion pipeline. The endpoint was introduced in v0.2.0 but could not produce valid images due to several compounding issues. All are resolved in this release.

---

## Bug Fixes

### ipadapter_faceid: end-to-end pipeline fixes

**Model switch — non-plus adapter required**

The default model in `llm_route.yml` was `ip-adapter-faceid-plus_sd15`, which requires two IP-Adapter heads: a FaceID projection head (InsightFace, 512-dim) and a CLIP appearance encoder. The diffusion server loads no CLIP encoder, so the second head received `None` embeddings and crashed in the UNet forward pass with `'NoneType' object has no attribute 'dtype'`.

The correct default model for a pure face-identity pipeline is `ip-adapter-faceid_sd15` (single head, FaceID only):

```yaml
ipadapter_faceid:
  implementation: diffusion_server
  model: ip-adapter-faceid_sd15   # was: ip-adapter-faceid-plus_sd15
  api_base: http://localhost:7860
```

**Face embedding shape**

InsightFace returns `normed_embedding` as a 1-D tensor of shape `[512]`. diffusers `validate_ip_adapter_image_embeds` requires 3-D tensors. The embedding is now shaped to `[1, 1, 512]` via two `unsqueeze` calls before further processing.

**CFG-compatible embedding (classifier-free guidance)**

Stable Diffusion runs with `guidance_scale > 1` by default, which activates classifier-free guidance. diffusers calls `.chunk(2)` on each embed to split negative and positive conditioning. A single `[1, 1, 512]` tensor only yields one chunk and raises `not enough values to unpack (expected 2, got 1)`.

The fix concatenates a zero tensor (negative) with the face embed (positive) along dim 0 to produce shape `[2, 1, 512]`:

```
ip_adapter_image_embeds=[torch.cat([zeros, faceid_embeds], dim=0)]  # [2, 1, 512]
```

**`image_encoder` dtype proxy**

When no CLIP encoder is loaded, `pipe.image_encoder` is `None`. diffusers still accesses `self.image_encoder.dtype` during embed preparation even for pre-computed embeds. A lightweight `_DtypeProxy` object is patched onto the pipeline after load to satisfy this access.

**Safety checker disabled**

SD v1.5's built-in safety checker was flagging face-conditioned outputs as NSFW and returning a black 1×1 PNG instead of the generated image. The safety checker is now disabled on pipeline load — this is a local inference server with no NSFW filtering requirement.

**MPS inference lock**

FastAPI runs synchronous route handlers in a thread pool. Concurrent requests raced on the same MPS pipeline, causing `Already borrowed` tensor errors. An `_infer_lock` now serializes all pipeline inference calls (both `/ipadapter` and `/ipadapter_faceid`).

---

### Retry logic: no retry on 4xx errors

The gateway's `retry_image_generation` (and `retry_text_completion`) previously retried on all exceptions, including HTTP 4xx client errors. A deterministic 400/422 response (e.g. "No face detected") would be retried 3 times, producing a burst of 3 identical failing requests in the diffusion server logs before the gateway returned 502.

4xx errors are now treated like timeouts — the retry loop exits immediately, because resending the same payload will never succeed.

---

### ipadapter_faceid: 422 with detail for face-detection failures

When no face is detected in the uploaded image, the diffusion server previously raised a bare `ValueError` caught as HTTP 400. This has been changed:

- A dedicated `NoFaceDetectedError` exception (subclass of `ValueError`) is raised in `pipeline.py`.
- The `/ipadapter_faceid` handler catches it and returns **HTTP 422 Unprocessable Entity** with a `detail` field:

```json
{"detail": "No face detected in the provided image"}
```

Error taxonomy for the diffusion server:

| Condition | HTTP status |
|---|---|
| Unknown model / wrong endpoint for model type | 400 Bad Request |
| Image bytes cannot be decoded | 422 Unprocessable Entity |
| No face detected in image | 422 Unprocessable Entity |
| Pipeline failed to load | 503 Service Unavailable |

---

### Error detail propagation

`impl_ipadapter.py` previously called `raise_for_status()` directly, which produces a generic `"400 Bad Request"` error with no context. The diffusion server's JSON `detail` field is now extracted and included in the raised `HTTPError`, so the actual reason (e.g. `"No face detected in the provided image"`) is visible in gateway logs and client error responses.

---

## Breaking Changes

None. The model name change in `llm_route.yml` is a correction — the plus variant was never functional with this server configuration.

- All v0.2.0 compatibility notes remain unchanged.
